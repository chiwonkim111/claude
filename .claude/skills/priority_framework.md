기능 우선순위를 결정할 때 아래 두 프레임워크를 함께 사용합니다.
- **MoSCoW**: 기능의 필수 여부를 4단계로 분류
- **RICE**: 수치 기반으로 우선순위 점수를 산출

---

## 1단계: MoSCoW 분류

먼저 각 기능을 아래 4가지 카테고리로 분류합니다.

| 분류 | 의미 | 판단 기준 |
|------|------|----------|
| **Must Have** | 반드시 있어야 함 | 없으면 서비스 자체가 불가능 |
| **Should Have** | 있어야 하지만 당장은 아니어도 됨 | 없으면 불편하지만 대안이 있음 |
| **Could Have** | 있으면 좋음 | 사용자 경험을 높이지만 핵심은 아님 |
| **Won't Have** | 이번 버전에서는 제외 | 나중에 고려하거나 영구 제외 |

### MoSCoW 판단 질문

기능마다 아래 질문에 답하여 분류를 결정합니다.

```
Q1. 이 기능 없이 서비스를 출시할 수 있는가?
    → NO  : Must Have 후보
    → YES : 다음 질문으로

Q2. 이 기능 없이도 핵심 가치를 전달할 수 있는가?
    → NO  : Should Have 후보
    → YES : 다음 질문으로

Q3. 이 기능이 사용자 만족도를 크게 높이는가?
    → YES : Could Have
    → NO  : Won't Have
```

### MoSCoW 매트릭스 출력 형식

```markdown
## 우선순위 매트릭스

### ✅ Must Have (MVP 필수)
| 기능 ID | 기능명 | 이유 |
|---------|--------|------|
| FR-001 | | MVP에 없으면 서비스 불가 |

### 🔵 Should Have (다음 스프린트)
| 기능 ID | 기능명 | 이유 |
|---------|--------|------|
| FR-002 | | 대안은 있지만 UX 저하 |

### 🟡 Could Have (여유 있을 때)
| 기능 ID | 기능명 | 이유 |
|---------|--------|------|
| FR-003 | | 있으면 좋은 부가 기능 |

### ❌ Won't Have (이번 버전 제외)
| 기능 ID | 기능명 | 이유 |
|---------|--------|------|
| FR-004 | | 리소스 대비 효과 낮음 |
```

## 2단계: RICE 점수 산출

Should Have 이상의 기능에 대해 RICE 점수를 산출하여 세부 순위를 정합니다.

### RICE 공식

```
RICE 점수 = (Reach × Impact × Confidence) / Effort
```

| 요소 | 설명 | 측정 기준 |
|------|------|----------|
| **Reach** (도달 범위) | 분기당 영향 받는 사용자 수 | 예상 사용자 수 (명) |
| **Impact** (영향도) | 개인 사용자에게 미치는 영향 | 3=매우 큼 / 2=큼 / 1=보통 / 0.5=작음 |
| **Confidence** (확신도) | 추정치에 대한 확신 정도 | 100%=확실 / 80%=높음 / 50%=보통 |
| **Effort** (노력) | 구현에 필요한 인원×개월 | 인원 × 개월 (person-month) |

### RICE 점수표 출력 형식

```markdown
## RICE 점수 산출표

| 기능명 | Reach | Impact | Confidence | Effort | RICE 점수 | 순위 |
|--------|-------|--------|------------|--------|-----------|------|
| 기능 A | 1,000 | 3 | 0.8 | 2 | 1,200 | 1 |
| 기능 B | 500 | 2 | 1.0 | 1 | 1,000 | 2 |
| 기능 C | 200 | 1 | 0.5 | 0.5 | 200 | 3 |

> RICE 점수가 높을수록 먼저 구현합니다.
```

---

## 최종 우선순위 결정 규칙

1. **Must Have**는 RICE 점수와 무관하게 최우선 구현합니다.
2. **Should Have**는 RICE 점수 내림차순으로 순위를 결정합니다.
3. **Could Have**는 리소스 여유가 있을 때 RICE 점수 순으로 검토합니다.
4. 점수가 비슷할 경우 **Effort(노력)가 낮은 기능을 먼저** 구현합니다 (Quick Win).
5. 우선순위 변경 시 이유를 반드시 `docs/PLAN.md` 변경 이력에 기록합니다.

---

## 우선순위 재검토 시점

아래 상황에서는 우선순위를 재검토합니다:

- 사용자 피드백으로 Pain Point가 변경되었을 때
- 기술적 제약이 새로 발견되었을 때
- 비즈니스 목표가 변경되었을 때
- 스프린트 종료 후 회고 시점
